## 6. 架构说明：能力模型 vs 接口模型

### 旧思路（接口列表）

```
GET /api/v1/calendar/list   ← 客户看不懂
POST /api/v1/calendar/create
PUT /api/v1/task/update
```

### 新思路（能力中心）

```
📅 日程能力
   ├─ 获取日程列表   → GET /api/v1/calendar/list
   ├─ 创建日程       → POST /api/v1/calendar/create
   ├─ 更新日程       → PUT /api/v1/calendar/{id}
   └─ 删除日程       → DELETE /api/v1/calendar/{id}

👤 用户能力
   ├─ 获取用户信息   → GET /api/v1/user/{id}
   └─ 同步用户       → POST /api/v1/user/sync

📋 项目能力
   ├─ 获取项目列表   → GET /api/v1/project/list
   ├─ 创建项目       → POST /api/v1/project/create
   └─ 删除项目       → DELETE /api/v1/project/{id}
```

**好处：**
- 客户、销售、实施说同一种语言
- 能力 = 收费单元，边界清晰
- 适配层不影响能力展示

## 7. 开放等级定义

| 等级 | 能力示例 | 收费模式 |
|------|---------|---------|
| **基础开放能力** | 用户查询、组织查询、日程CRUD、任务CRUD | 包含在产品授权内 |
| **高级开放能力** | Webhook推送、批量同步、扩展字段、字段映射 | 高级版授权 |
| **定制集成** | SAP集成、ERP对接、MES打通、特殊接口 | **实施项目收费** |

### 销售话术

> "基础接口能力包含在授权里；深度集成（SAP/MES等）属于实施项目，按需评估人天。"

---

## 8. 适配层架构（三层隔离）

```
客户系统（ERP/MES/WMS）
        ↓
客户适配层（Adapter）- 按客户独立部署，项目承担，不进产品
        ↓
标准能力层（日事清 API）
        ↓
核心业务服务
```

### 关键设计原则

1. **能力不变，接口可以变** — 客户看到的永远是"能力"，但底层适配层可以定制
2. **适配层独立版本** — 客户A用v2.3，客户B用v3.1，互不污染
3. **定制接口不进产品** — 客户的SAP映射、ERP字段等全部在适配层
4. **扩展字段预留** — 标准API预留 `_extends` 字段，适配层注入

## 10. 在线调试（Try it Out）

### 功能点

- 输入 Token（自动填充）
- 填写请求参数
- 一键发送请求
- 实时查看响应

### 价值

- 客户自测率提升 → 减少"客户→项目经理→开发"的流转
- 80%的调试问题客户自己解决

## 11. 风险提示

| 风险 | 影响 | 应对 |
|------|------|------|
| 能力模型设计不合理 | 后续扩展困难 | 第一期先定义5-8个核心能力，后续扩展 |
| 适配层失控 | 主代码污染 | 严格禁止定制进入产品层 |
| 客户要求开放底层接口 | 系统安全风险 | 定义"开放能力"边界，不是"开放系统结构" |
| 多认证适配复杂度 | 项目超期 | 第一期预留架构，第二期按需实现 |
